GpsGate Data Pipeline - Project Handover

Project Overview

The GpsGate Data Pipeline is a Flask-based ETL system that extracts fleet tracking data from the GpsGate API, processes it, and stores it in a PostgreSQL database for business intelligence purposes. The system handles 8 different event types and maintains both fact and dimension tables.

Production URL: https://powerbixgpsgatexgdriver.onrender.com

System Architecture

Core Components

Web Application (application.py)

Event Types Processed

  1. Trip - Vehicle trip records
  2. Speeding - Speed violation events
  3. Idle - Vehicle idle time tracking
  4. AWH - After work hours usage
  5. WH - Work hours tracking
  6. HA - Harsh acceleration events
  7. HB - Harsh braking events
  8. WU - Weekend usage patterns

Database Schema

API Configuration

All event types use standardized configuration that avoids server state issues:

  • Report ID: 25 (for all events including Trip)
  • Tag ID: 39
  • App ID: 6

This configuration prevents the "reportId: 0" server errors that were encountered previously.

Operational Procedures

Scheduled Operations

Daily Sync (2:00 AM UTC)

  • Syncs dimension tables from API
  • Processes yesterday's fact data
  • Handles 8 event types sequentially
  • Records execution status

Weekly Backfill (Sunday 3:00 AM UTC)

  • Processes last 7 days of data
  • Reconciles any missing records
  • Handles data quality issues
  • Full fact table refresh

Manual Operations

Dashboard Access: /dashboard/

  • Table record counts (fact and dimension)
  • Manual sync triggers
  • Job execution monitoring
  • Scheduler status tracking

Manual Triggers:

  • Dimension Sync: Updates all dimension tables from API
  • Fact Sync: Processes fact data for specified date range
  • Full Backfill: Complete dimension + fact sync

Data Flow

  1. Render Phase: Submit request to GpsGate API
  2. Result Phase: Retrieve and cache API response
  3. Download Phase: Process CSV data
  4. Transform Phase: Clean and validate records
  5. Load Phase: Insert to database with duplicate handling

Troubleshooting

Common Issues

"reportId: 0" Error

  • Cause: GpsGate server state issue
  • Solution: Wait 10-15 minutes or use report_id=25 configuration
  • Prevention: Current config avoids this issue

Missing Records

  • Check: job_executions table for failed runs
  • Action: Use manual backfill for specific date ranges
  • Tool: Dashboard manual triggers

Cron Job Failures

  • Check: Render service logs
  • Common: Python path or import issues
  • Fix: Verify startCommand includes full path

FAQ & Onboarding Roadmap

Below are direct answers to common operational questions and a concise step-by-step onboarding roadmap for adding new customers or events.

1) Adding extra events for specific customers

  • Where to change: EVENT_CONFIG in these files: dashboard_enhanced.py, scheduled_daily_sync.py, scheduled_weekly_backfill.py.
  • Steps:
    1. Add the new event entry to EVENT_CONFIG (use report_id=25 by default for reliability).
    2. Create a new Fact model in models.py following existing fact table patterns and add required columns.
    3. Add a migration (Alembic/Flask-Migrate) to create the table and any indexes.
    4. If CSV/response parsing differs, update parsing logic in data_pipeline.py to handle the new response_key.
    5. Update the dashboard UI (HTML in dashboard_enhanced.py) to display the new stat card and include the table in get_table_counts().
    6. Test with a 1-day backfill and manually verify database rows and dashboard counts.

2) Passing dynamic Report ID / Tag ID / App ID per customer

  • Recommended approach: store per-customer configuration in the database.

    • Create table customer_configs: customer_id, app_id, token, base_url, tag_id, event_map (JSON), active.
    • Scheduled jobs (scheduled_daily_sync.py, scheduled_weekly_backfill.py) iterate active customers and pass their config into process_event_data.
    • Dashboard/manual endpoints accept optional override fields (app_id, report_id, tag_id, token) that take precedence over DB config.

  • Alternative for ad-hoc runs: pass app_id/token/tag_id directly in the API payload to the manual trigger endpoints.

3) Handling waiting time for report generation

  • Flow: render request -> poll result endpoint -> when ready download CSV -> transform -> load.
  • Best practices:
    • Implement polling with backoff (e.g., 5s, 15s, 30s) up to a sensible timeout (e.g., 10 minutes).
    • Cache render and result rows in render/result tables to avoid duplicate requests.
    • If API returns reportId: 0 or similar server-state errors, mark job as degraded and retry later or use fallback report IDs.
    • Log render start/end timestamps and durations for observability.

4) Onboarding a new customer — step-by-step roadmap

Prerequisites: GpsGate credentials for the customer (token), example app_id, report_id, tag_id, access to the production DB for configuration.

Step-by-step:

  1. Create customer configuration (DB row)

    • Insert into customer_configs:
      • customer_id: short id
      • app_id, token, base_url, tag_id
      • event_map: JSON mapping of event names -> {report_id, event_id, response_key}
      • active: true

  2. Test API connectivity

    • Quick render test for a small date window using internal /render flow or Postman with customer credentials.
    • Confirm result becomes available and CSV is downloadable.

  3. Initial dimension sync

    • Run dimension sync for the customer (it uses the shared dimension sync unless per-customer variations are required):
      python sync_dimensions_from_api.py
    • Verify dim_ table counts and sample data.

  4. Short fact sync (smoke test)

    • Run a 1-day fact sync using the dashboard manual trigger or internal backfill script with payload override for the customer.
    • Example: trigger /dashboard/trigger/fact-sync with body including app_id, token, tag_id.

  5. Verify results

    • Check GET /dashboard/stats/table-counts and GET /dashboard/status/recent.
    • Spot-check data quality and timestamps.

  6. Enable scheduled runs

    • If the customer should be included in the existing daily/weekly jobs, ensure their active flag is set.
    • For custom schedules, add a dedicated cron job in render.yaml or an external scheduler.

  7. Document the onboarding entry (internal tracker)

    • Add credentials, test results, any anomalies, and the date of activation.
